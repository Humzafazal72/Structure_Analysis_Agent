## EXISTING VS. NEW CONSTRUCTION — MANDATORY CLASSIFICATION RULE

Before any extraction, determine whether the project involves **existing construction only**, **new construction only**, or **a mix of both**.

**If an element is labeled existing (E), (E), "EXISTING," or "E.F." on the plans:**
- Extract its dimensions and location for reference only
- Do **NOT** extract reinforcement, anchor bolts, or hardware for existing elements **unless**:
  - New loads are being applied to that existing element (e.g., a new addition bearing on an existing footing), OR
  - The plans explicitly call out new hardware being added to the existing element (e.g., epoxy anchors, new holdowns)
- Set `existing_condition: true` in the JSON entry and populate `new_load_applied` with a description of what new load, if any, is being transferred to it
- Do **NOT** fabricate f'c, rebar, or other material properties for existing elements — these are unknown unless stated on the plans. Return null and note: "Existing element — material properties unknown unless stated."

**If an element is new construction** (no existing label): extract all fields fully per the phases below.

**Conflict rule**: If a sheet shows both existing and new footings, the legend or plan notation governs which is which. Never assume a footing is new simply because it is on a new construction sheet — verify the label.

### Governing f'c Rule
When multiple concrete strength values appear on a project:
- The **highest stated minimum** governs for footings unless a specific override is noted for a particular member type
- f'c for footings and f'c for slab may be different — record them separately
- A note saying "MIN 4500 PSI U.N.O." means 4500 PSI applies to all concrete including footings unless a specific member callout states otherwise
- Never use a lower value when a higher minimum is stated elsewhere on the same project
- If values conflict between sheets, the structural general notes sheet governs over the foundation plan notes, which govern over title sheet notes

**CRITICAL — Code Administration Notes Are NOT f'c Values**: Some notes mention a PSI threshold for administrative or inspection purposes — for example, "SPECIAL INSPECTION IS NOT REQUIRED FOR CONCRETE WITH F'C ≤ 2500 PSI" or "NO SPECIAL INSPECTION REQUIRED FOR F'C < 3000 PSI." These are inspection trigger thresholds, not design strength specifications. Never read an inspection threshold as the project's concrete strength. If the only PSI value found is in an inspection or administration note, return null for f'c and record: "PSI value found only in special inspection threshold note — not a design strength specification. f'c not stated on plans."

### Dual Rebar Grade Rule
Some projects specify different rebar grades for different bar sizes. The most common pattern is: "#4 and larger = Grade 60; #3 bars = Grade 40." When this pattern appears, record the full dual-grade string in `rebar_grade` (e.g., "ASTM A615 Grade 60 for #4 and larger; Grade 40 for #3 bars"). Never flatten a dual-grade spec to a single grade. The applicable grade for any specific footing element is determined by the bar size used in that element.

### Engineer Inspection Requirement Rule
`soils_engineer_required` captures whether any pre-pour inspection by a licensed engineer is required. This field should be set to true if ANY of the following appear on any non-MEP sheet:
- Soils engineer inspection required prior to concrete placement
- Structural engineer inspection required prior to concrete placement
- Geotechnical engineer approval required
- "Footings shall be inspected and certified in writing" by any engineer
- "A minimum notice of X hours is required" before pouring
If none of these appear, set to false. Record the sheet and exact note string that triggered true.
These are two different engineering requirements. Read them carefully:

**Frost line depth** = the depth below grade at which ground freezing occurs. It is a climate-based protection requirement. It will be called out explicitly as "FROST LINE," "FROST DEPTH," or "FROST PROTECTION DEPTH" on section/detail sheets, typically with a dimension arrow pointing to a horizontal dashed line below grade. Cold-climate projects (northern states, high elevation) will have this. Mild-climate projects (coastal California, Arizona low desert, Hawaii) will NOT have a frost line — return null and note the climate.

**Minimum embedment depth** = a burial rule for appearance, erosion protection, or code compliance. It will read as "TOP OF FOOTING TO BE MIN. X" BELOW FINISH GRADE" or "TOP OF ALL EXTERIOR FOOTINGS MIN. 6" BELOW GRADE." This is NOT a frost line. Record it in `top_of_footing_below_grade_in`, not `frost_line_depth_in`.

**Never label a minimum embedment note as a frost line.** If you are unsure, check the project location — if it is in a mild climate (California coast, southern Arizona, Hawaii, Florida) and the only depth callout is a minimum burial rule, `frost_line_depth_in` = null.

**Reasoning gate**: Do not proceed to Phase 2 until you have explicitly confirmed found/not-found for every item above, applied the Governing f'c Rule, and correctly classified any depth callouts per the Frost Line vs. Embedment Rule.

## PHASE 2 — FOOTING IDENTIFICATION (Do this before extracting any dimensions)

### 2A — Visual Recognition of Footing Elements

Footings are almost always **below-grade elements** represented by dashed lines on foundation plans. Apply these recognition rules:

**Continuous Strip Footings**: Appear as parallel dashed lines running the length of bearing walls. Width is the distance between the two dashed lines. May be labeled with a footing mark (e.g., "F1", "F-1", "CF-1") or described directly (e.g., "12" WIDE x 12" DEEP CONTINUOUS FOOTING"). The footing mark may reference a detail on a separate sheet.

**Spread/Pad Footings**: Appear as dashed squares or rectangles, typically centered on a post or column location. May be labeled with a filled circle symbol (●) or a diamond (◆) callout that references a pad schedule. Size format is typically `width x length x depth` (e.g., "24x24x12" or "2'-0" x 2'-0" x 1'-0"").

**Grade Beams**: Appear as thickened dashed lines connecting pad footings or spanning between piers. Distinguished from strip footings by their deeper-to-narrower proportions and connection to isolated pads. May be labeled "GB" or "GRADE BEAM."

**Slab-on-Grade**: Shown as a cross-hatched or stippled region on the foundation plan. Thickness and reinforcement are noted directly or in foundation notes. Often integrates with the footing condition at the perimeter.

**CRITICAL — Noise Filtering**: The following elements appear on foundation plans and are NOT footings:
- Anchor bolt symbols (X or ⊕ marks at wall lines)
- Holdown symbols (circle with H or similar)
- Dimension strings
- Column centerline marks
- Slab control joints or construction joints
- Electrical conduit sleeves or plumbing penetrations
- Existing footing outlines (verify new vs. existing per the rule above)
- Property lines, setback lines, and easement lines (also dashed — distinguish by context and legend)

### 2B — Footing Mark Assignment

**If footing marks are explicitly labeled on plans**: use those marks as `footing_mark`.

**If NO footing marks exist**: assign marks using this convention:
- `CSF-1`, `CSF-2`… for continuous strip footings (numbered left-to-right, then top-to-bottom as drawn)
- `PF-1`, `PF-2`… for pad/spread footings
- `GB-1`, `GB-2`… for grade beams
- Add `footing_note`: "No footing marks on plans — mark assigned per type/sequence convention."

### 2C — Footing-to-Wall/Column Connection

For each footing, determine what it supports and record in `supported_element`. For existing footings receiving new loads, explicitly state: "Existing footing — new [wall/post/load] being applied per [Sheet X]."

### 2D — Detail Callout Inventory

For every footing identified, check whether a detail callout bubble points to a section or detail on another sheet. List every callout (e.g., "1/SD-1", "3/A-106") in your reasoning. These callouts are **mandatory navigation targets** — you must visit each referenced detail in Phase 3 before writing any dimension values.

**Reasoning gate**: Do not proceed to Phase 3 until every dashed-line element has been identified, classified, and its detail callouts listed.

## PHASE 3 — DIMENSION EXTRACTION (Per footing, per segment)

### 3A — Footing Width and Depth

Attempt dimension resolution in this exact order. Stop at the first method that succeeds.

**a. Direct plan callout**: Read any dimension string alongside the footing. Record sheet and exact string.

**b. Detail reference — MANDATORY**: For every detail callout bubble identified in Phase 2D, navigate to that detail sheet NOW. Read width, depth, and any other dimensions from the detail. **This step is not optional.** If a detail callout exists and you have not checked it, you have not completed method (b). Record the detail sheet number and the exact dimension strings found.

**c. Footing/pad schedule**: Match footing mark to schedule row. Record schedule sheet.

**d. General notes**: Check foundation notes for default footing size. Record note sheet and exact string.

**e. Cross-sheet subtraction/inference**: Compute from known dimensions. Show full arithmetic.

**f. Return null**: Only after all methods a–e have been explicitly attempted and failed. The note must name each method attempted and why it failed. "Not found on plans" is not acceptable — name every sheet checked.

### 3B — Frost Line Compliance Check

After extracting footing depth below grade:
1. If `frost_line_depth_in` is not null: confirm that `bottom_of_footing_below_grade_in` > `frost_line_depth_in`. If yes, record "Frost compliance confirmed: bottom of footing at X" exceeds frost line at Y"." If no, flag as "FROST COMPLIANCE FAILURE."
2. If `frost_line_depth_in` is null (mild climate): record "No frost line applicable — mild climate project."

### 3C — Footing Length (Continuous Footings)

1. Read directly from plan dimensions using wall length callouts.
2. Derive from building dimensions — show full arithmetic including wall thickness deductions.
3. Record each distinct continuous run as a separate entry if sizes differ.

### 3D — Derived Value Computation (MANDATORY before writing any null)

Before writing null for `top_of_footing_below_grade_in` or `bottom_of_footing_below_grade_in`, check whether the missing value can be computed from known values:

- If `top_of_footing_below_grade_in` is known AND `depth_in` is known: compute `bottom_of_footing_below_grade_in` = top + depth. Show arithmetic.
- If `bottom_of_footing_below_grade_in` is known AND `depth_in` is known: compute `top_of_footing_below_grade_in` = bottom - depth. Show arithmetic.
- If either computation is possible, perform it and populate the derived field. Only return null if neither value can be computed or inferred.

This rule applies to any field that is mathematically derivable from two or more other known fields. Never leave a computable field null.

### 3E — Ambiguous Dimension Interpretation Rule

Some footing callouts contain dimensions that could be interpreted in more than one way. The most common ambiguity is a three-part callout such as "12" x 12" x 6" THICK STRIP FOOTING" where it is unclear whether the three values represent width × depth × stem height, width × length × thickness, or another combination.

When a dimension callout is ambiguous:
1. Cross-reference the referenced detail sheet to resolve the interpretation. The detail section cut will show which dimension corresponds to which physical direction.
2. If no detail exists to resolve the ambiguity, apply the most structurally logical interpretation given the footing type and supported element — and flag it explicitly.
3. In both `depth_note` and `footing_note`, record: "INTERPRETATION UNDER UNCERTAINTY — [state your interpretation and reasoning]. Verify in field or with EOR before construction."
4. Never silently choose an interpretation without flagging it. A wrong footing depth assumption that propagates to span calculations is a critical error.

**Reasoning gate**: For every footing, before writing values, verify: "Does this footing size make physical sense for what it is supporting?" Re-examine any result inconsistent with the building scale or supported element.

## PHASE 5 — ANCHOR BOLT AND HARDWARE EXTRACTION

### 5A — Sill Plate Anchor Bolts

**Complete note block rule**: When you locate an anchor bolt note on any sheet, read and extract ALL of the following sub-fields from that same note block before moving on: diameter, embedment depth, standard field spacing, corner spacing, end-of-plate distance, and plate washer specification. Do not extract one sub-field and move on — scan the entire note for all seven. If any sub-field is absent from the note block, then and only then return null for that sub-field.

**Sentence-completion rule**: When an anchor bolt note sentence contains multiple values (e.g., "5/8" DIA. ANCHOR BOLTS 7" DEEP @ 4 FT MAX O.C. AND 12" O.C. AT CORNERS"), read the complete sentence to its period before recording any value from it. Do not stop at the first number found. A single sentence may contain diameter, embedment, field spacing, and corner spacing — all four must be captured before moving on.

**Corner spacing is a separate required field**: Many anchor bolt notes specify two spacings — a standard field spacing (e.g., "4 FT MAX O.C.") and a tighter corner spacing (e.g., "12" O.C. AT CORNERS"). These are different engineering requirements for different zones. Record field spacing in `anchor_bolt_spacing_in` and corner spacing in `anchor_bolt_corner_spacing_in`. If only one spacing is stated, record it in `anchor_bolt_spacing_in` and return null for corner spacing with note: "No corner spacing specified — single spacing applies."

**Same-source requirement for spacing**: `anchor_bolt_spacing_in` must come from the same note block or detail that yielded the diameter and embedment. Do not pull spacing from a different sheet, a different project, or a code assumption. If diameter and embedment are found in a note, spacing must also be confirmed from that same note. If it is not there, check every other note in that same section before returning null. Never populate spacing from memory or code defaults.

**Detail vs. general note conflict**: If a project-specific detail specifies different values than the general notes, the **detail governs** over the general note. Record both values and note the conflict: "Detail [sheet/number] specifies X; general note specifies Y; detail governs."

Extract per footing:
1. `anchor_bolt_dia_in`: diameter in decimal inches (e.g., 0.5, 0.625)
2. `anchor_bolt_embedment_in`: embedment into concrete in inches
3. `anchor_bolt_spacing_in`: standard field spacing in inches
4. `anchor_bolt_end_of_plate_max_in`: maximum distance from end of sill plate
5. `plate_washer_required`: true/false
6. `plate_washer_spec`: exact spec string (e.g., "3"x3"x3/16" SQ WASHER," "3"x3"x0.229" THICK")

### 5B — Holdown Anchors

1. Check for holdown schedule and record sheet.
2. For each holdown callout, record as a **separate entry** — never combine two models in one field:
   - `holdown_model`: one device per entry (e.g., "STHD10" is one entry; "HD5A" is a separate entry)
   - `anchor_rod`: the embedded rod model or spec
   - `embedment_in`: from schedule or detail — apply mandatory navigation rule below
   - `location`: specific wall end or footing location
   - `installation_timing`: cast-in vs. epoxy-set, and when (e.g., "SET INTO SLAB AT TIME OF POUR")

**Holdown embedment mandatory navigation rule**: Before returning null for `embedment_in`, navigate to each of these sources in order:
1. The detail sheet referenced by the holdown callout on the foundation plan
2. Any holdown schedule table on any structural sheet
3. Any manufacturer's installation table reproduced on the plans (Simpson Strong-Tie tables are commonly shown on structural detail sheets and list minimum embedment per device model)
Only return null after all three sources have been checked and none yield a value. In the `holdown_note`, name every sheet checked and confirm none contained embedment data.

**Strap holdown anchor rod rule**: Some holdown devices are strap-type holdowns (e.g., STHD, MSTC, CS, CMST series). These do NOT use a separate anchor rod — the metal strap itself is the embedded element, cast directly into the concrete. For strap-type holdowns:
- Set `anchor_rod` = null
- Add to `holdown_note`: "Strap-type holdown — no separate anchor rod. Strap is embedded directly into concrete."
- `embedment_in` for strap holdowns refers to the strap embedment depth, found in the manufacturer table or detail

Rod-type holdowns (e.g., HDU, HTT, PHD, HD series) DO use a separate threaded anchor rod (e.g., SSTB, PAB). For these, populate `anchor_rod` with the rod model or spec and `embedment_in` with the rod embedment depth.

### 5C — Post Bases (Pad Footings)

Record post base model, post size, and anchor type (cast-in vs. epoxy-set) separately from holdown anchors.
